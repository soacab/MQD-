# CheckFlow 前端三方字段对账

本文档记录 `企业办公_点检AI方案设计.md` 第四章、`CheckFlow_原型.html` 和当前前端/API 的差异。实现时不擅自展示未确认字段。

## 对账原则

- 第四章定义功能、权限、流程和业务规则。
- 原型定义页面结构、字段呈现、按钮和交互形态。
- 当前接口和前端是实现现状，不作为产品字段最终依据。
- 任一来源冲突时，先标记为待确认差异，再由用户确认。

## 第四章有但原型没有

- 4.1 后台设置的最近保存时间、最近保存人、未保存修改状态：当前接口只暴露最小设置值，原型未完整落位。待确认是否需要本轮补 UI 占位。
- 4.4 多候选文件待选态、手动补充 VDrive 文件路径、系统直连多机型明细：第四章规则完整，但当前 P0 接口仍是 mock 或未接入。待真实扫描/直连接口后再进入正式 UI。
- 4.7 报告更正、PDF 导出、节点多轮过程展开：第四章明确要求，当前 P0 未完整实现。页面仅保留边界提示，不做假入口。

## 原型有但第四章没写

- 顶栏品牌、紧凑导航、状态卡片密度、部分视觉标签属于原型补充表达。当前重设计保留视觉表达，但不因此新增业务字段。
- 原型中的部分演示数据用于展示布局，不作为接口字段依据。

## 接口/页面有但两者都没有

- `project_id`、内部任务 ID、规则快照 JSON、部分内部状态码属于实现字段。除调试或错误定位外，不作为主要 UI 字段展示。
- 当前页面曾直接暴露 QG 节点 ID、MQ 人员 ID 等内部 ID。重设计中优先展示名称；接口仍使用 ID 传参。
- `archive_ready_count` 是当前 dashboard API 的实现统计，第四章工作台只定义进行中、复查中、待跟进三分区。当前 UI 将其作为治理提示，不扩展成新业务分区。

## 命名不同但可能同义

- 点检任务 / 检查任务：统一为“点检任务”。
- 项目删除 / 项目作废 / 项目隐藏：第四章语义是不物理删除的作废/隐藏，UI 使用“作废/隐藏”，接口仍调用现有删除 helper。
- 报告 / 检查档案 / 历史报告：列表入口统一为“检查档案”，具体弹窗或详情称“报告”。
- MQ 人员 / 归属工程师 / 点检工程师：按页面语境展示，字段语义待业务最终统一。

## 本轮 UI 处理

- 页面字段和操作只展示能追溯到第四章或原型的内容。
- 待确认差异以 `field-governance-note` 提示或本文档记录承载。
- 不把当前接口多余字段包装成产品功能。
