# Product

## Register

product

## Users

CheckFlow 面向 MQD 工程师、规则管理员、项目管理员和权限管理员。用户在收到线下 QG 点检任务后进入系统，完成任务创建、检查项确认、归档、整改跟进、复查和历史报告查阅。

## Product Purpose

CheckFlow 用于把 MQD QG 点检中的文件查找、状态记录、报告归档、整改跟进和规则版本管理结构化。系统不替代工程师判断，工程师仍是最终结论确认人。

## Brand Personality

克制、清晰、可信。界面应像工作台工具，不像营销页面；优先让用户快速理解当前项目、当前节点、当前轮次和下一步动作。

## Source Of Truth

`企业办公_点检AI方案设计.md` 第四章定义功能、权限、流程和业务规则。`CheckFlow_原型.html` 定义页面结构、字段呈现和交互形态。当前接口和前端是实现现状，不是产品字段清单的最终依据；接口为实现现状。

当第四章、原型和接口不一致时，先记录为待确认差异，不在 UI 中擅自新增字段或功能。

## Anti-references

不要把接口中存在但第四章和原型都未出现的字段直接展示给用户。不要把项目创建做成独立用户入口。不要用大面积装饰、营销式卡片或过度圆角削弱工具感。

## Design Principles

- 产品依据优先于接口现状：第四章和原型共同约束字段、术语和流程。
- 工程师最终判断优先：AI、系统直连和自动检查结果只提供初判，最终结论必须清晰归属。
- 状态分区优先：进行中、复查中、待跟进、检查档案和规则配置应在信息架构上分清。
- 只展示已确认语义：待确认差异以治理提示和文档记录承载，不伪装成正式功能。
- 小步验证：前端重设计不得破坏现有 P0 API 调用链路。

## Accessibility & Inclusion

默认按 WCAG AA 对比度目标设计。状态不能只靠颜色表达，需同时使用文字标签。动画只用于状态反馈，不做装饰性加载编排。
