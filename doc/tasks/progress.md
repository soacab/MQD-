# CheckFlow MVP 任务进度

## 总体说明

本文件是 CheckFlow MVP 当前进度的总入口，用于对齐 `doc/tasks/` 模块任务、现有代码事实和 `企业办公_点检AI方案设计.md` 中的实际业务需求。

注意：本文件是当前状态总入口。执行具体模块前仍需读取对应任务文件；若模块 checklist 与本文件冲突，应以本文件的状态口径和最新验证结果为准，并在后续小任务中同步修正。

## 状态口径

| 状态 | 含义 |
| --- | --- |
| ✅ 已验证可用 | 已有实现和验证，能支撑当前定义的业务或工程目标。 |
| 🟡 骨架/最小链路已通，但未完全对齐业务方案 | 已有接口、页面或测试骨架，但距离实际业务流程、交互和验收标准仍有缺口。 |
| ⏳ 未开始 | 尚未进入实现，或只有任务文档无可用实现。 |
| ⏸ 暂缓 | 设计上确认暂不进入当前 MVP 优先级。 |

## 当前事实

- 后端 P0 主链路已有 API 和 unittest 覆盖，可以演示从项目、规则、任务、检查项确认、归档、报告、整改到复查的最小闭环。
- Phase 1 已明确保留轻量 SQL helper 路线；表结构、迁移、seed、集中枚举、核心唯一约束测试和 Alembic offline SQL 生成已验证可用，SQLAlchemy ORM Model 不作为当前 MVP Phase 1 验收项。
- 后端实现目前集中在 `backend/app/main.py`，后续继续扩展前应考虑按领域拆分服务、路由和仓储，避免主文件继续膨胀。
- 自动检查当前仍是 mock：`file_existence` 会写入模拟结果，不代表真实 VDrive 扫描、候选文件匹配或异常降级已经完成。
- 前端 P0 页面已接入登录、后台、项目、规则、点检、报告、整改复查等现有业务 API，可以支撑最小人工操作闭环。
- `企业办公_点检AI方案设计.md` 中的工作台三分区、候选文件确认、完整三栏点检体验、归档弹窗、整改通知文案、复查范围弹窗、复杂检查档案筛选等体验仍需继续补齐。

## Phase 2 复核结论

Phase 2 当前不是 ✅ 完成，而是 🟡 最小链路可用、完整验收未完成。`doc/tasks/02-auth-user-permission.md`、`doc/tasks/03-project-management.md`、`doc/tasks/04-vdrive-link-validation.md` 中的历史 checklist 暂不能作为完成依据，当前真实状态以本文件为准。

- 已具备：项目创建、VDrive mock 校验、项目列表和详情、加单、软删除，以及 `deleted` 项目阻断创建点检任务，能够支撑 P0 最小主流程。
- 未完整：用户权限前端仍缺权限更新、启用、停用、删除访问配置、权限态导航和无权限提示。
- 未完整：项目删除确认仍是前端直接提交项目名，不是真实用户输入确认；项目历史信息回填和编辑体验也未完整。
- 边界说明：VDrive 当前仍是 mock URL 解析和模拟返回，只代表 MVP mock 边界，不代表真实路径可访问校验已经完成。

## 前端与业务方案对齐缺口

- 权限后台已接入最小 API，但仍需补停用/启用、删除访问配置、权限态导航和更明确的无权限提示。
- 项目与任务创建已支持最小表单和 VDrive 校验，但仍需补历史信息回填、QG 节点推荐、两步确认和创建后自动跳转策略。
- 点检执行页已支持检查项导航、详情、最终确认和转人工，但仍需补候选文件确认、系统直连详情、完整三栏状态分组和归档弹窗。
- 报告页已支持列表和详情，但仍需补完整检查档案筛选、逐轮过程明细、报告更正入口、原始结论和更正后结论的追溯关系。
- 整改页已支持单项标记、撤销、待跟进落实和触发复查，但仍需补整改通知文案、批量操作、复查范围确认和二次确认。
- 工作台需作为 P1 高优先级，展示进行中、复查中、待跟进、待归档入口，并支持从待办跳转到任务、检查项、整改项或报告。

## 建议下一阶段顺序

1. 补齐 P0 前端体验细节：权限态导航、任务创建向导、归档弹窗、整改通知、复查范围确认。
2. 建设工作台与查询，形成进行中、复查中、待跟进、待归档入口。
3. 推进报告更正和更完整的检查档案详情。
4. 最后推进 VDrive 扫描与文件存在性自动检查。

## 使用规则

- 每次实现模块前，先读取对应模块文件的目标、前置依赖和验收标准，同时以本文件判断当前优先级。
- 每完成一个子任务，更新对应模块文件中的 checklist；若该 checklist 与本文件状态冲突，应同步修正或在本文件记录原因。
- 每完成一个模块，必须更新本文件中的后端/API 状态、前端/交互状态、业务方案对齐结论和下一步。
- P2 暂缓模块只有在 P0/P1 验收通过后再开始，或由用户明确指定提前启动。
