# CheckFlow MVP 任务进度

## 总体说明

本文件是 CheckFlow MVP 当前进度的总入口，用于对齐 `doc/tasks/` 模块任务、现有代码事实和 `企业办公_点检AI方案设计.md` 中的实际业务需求。

注意：本文件是当前状态总入口。执行具体模块前仍需读取对应任务文件；若模块 checklist 与本文件冲突，应以本文件的状态口径和最新验证结果为准，并在后续小任务中同步修正。

## 状态口径

| 状态 | 含义 |
| --- | --- |
| ✅ 已验证可用 | 已有实现和验证，能支撑当前定义的业务或工程目标。 |
| 🟡 骨架/最小链路已通，但未完全对齐业务方案 | 已有接口、页面或测试骨架，但距离实际业务流程、交互和验收标准仍有缺口。 |
| ⏳ 未开始 | 尚未进入实现，或只有任务文档无可用实现。 |
| ⏸ 暂缓 | 设计上确认暂不进入当前 MVP 优先级。 |

## 当前事实

- 后端 P0 主链路已有 API 和 unittest 覆盖，可以演示从项目、规则、任务、检查项确认、归档、报告、整改到复查的最小闭环。
- Phase 1 已明确保留轻量 SQL helper 路线；表结构、冻结的 Alembic 初始迁移、seed、集中枚举、核心唯一约束测试和 Alembic offline SQL 生成已验证可用，SQLAlchemy ORM Model 不作为当前 MVP Phase 1 验收项。
- 后端实现已初步拆分到 `backend/app/api/routers/` 与 `backend/app/services/`，后续继续细化仓储/领域边界，避免服务层继续膨胀。
- 自动检查任务创建主路径仍保留 mock result 以兼容 P0；后端已新增检查项级 VDrive 候选文件扫描、候选保存、选择和重试 API，真实 VDrive 联调与前端候选确认界面仍未完成。
- 权限审查发现的业务数据范围 RBAC 已补齐并通过后端测试验证：`inspection_engineer` 仅能访问本人负责的项目及关联点检任务/派生数据（以 `projects.mq_user_id` 为准；无 MQ 人员的历史数据按任务创建人兼容），`project_admin` 可访问全量业务数据，组合权限取并集，权限管理不默认获得业务范围。
- 2026-07-01 修复复核：后端新增覆盖重复确认、历史报告读取、重复作废、作废原因、撤销/关闭审计和新建任务权限口径的测试；前端新增覆盖新建任务按钮禁用态、复查跳转口径和历史字段回显的结构测试。前端当前未配置 ESLint，`npm run lint` 不是验收命令。
- 前端 P0 页面已接入登录、后台、项目档案、规则、点检、报告、整改复查等现有业务 API，可以支撑最小人工操作闭环；完整体验和真实自动检查仍需继续补齐。
- `企业办公_点检AI方案设计.md` 中的工作台分区已接入最小待办入口；候选文件确认、完整三栏点检体验、归档弹窗、整改通知文案、复查范围弹窗、复杂检查档案筛选等体验仍需继续补齐。

## Phase 2 复核结论

Phase 2 当前不是 ✅ 完成，而是 🟡 最小链路可用、完整验收未完成。`doc/tasks/02-auth-user-permission.md`、`doc/tasks/03-project-management.md`、`doc/tasks/04-vdrive-link-validation.md` 中的历史 checklist 暂不能作为完成依据，当前真实状态以本文件为准。

- 已具备：新建点检任务时自动创建或复用内部 Project、VDrive mock 校验、项目档案列表和详情、加单、软删除，以及 `deleted` 项目阻断创建点检任务，能够支撑 P0 最小主流程。
- 已接通：用户权限后台支持权限更新、启用、停用、删除访问配置、权限筛选和无权限只读提示；“权限管理”不再自动获得点检、规则或项目管理能力。
- 已补齐：业务数据范围 RBAC 已覆盖本人负责范围（项目 MQ 人员口径）、项目管理全量范围、组合权限取并集和权限管理无业务范围。
- 已收敛：项目不再作为普通用户独立创建入口，工作台顶栏的新建任务弹窗是唯一用户侧新建点检任务入口；项目档案页仅用于历史基础信息维护、加单和作废。无点检执行权限用户可见新建按钮但不可点击。
- 未完整：真实 VDrive 可访问性校验仍未完整。
- 边界说明：VDrive 当前仍是 mock URL 解析和模拟返回，已有可替换 Adapter 边界和任务创建前校验动作，但不代表真实路径可访问校验已经完成。

## 与五份核心设计文档的主要差距

- 架构差距：后端已初步拆分到 routers/services，并形成 Project、Rule、Inspection、Report 等服务边界；仓储和领域边界仍需继续细化。
- API 差距：VDrive 候选文件识别、候选文件选择/列表和自动检查重试后端接口已补基础；报告更正和报告导出等接口仍缺失，部分已实现接口路径与设计稿仍有差异。
- 状态机差距：自动检查后端已补 `candidate_waiting`、not_found、多候选、扫描失败、重试等基础状态和业务动作；任务创建时仍保留 mock success 主路径，前端未形成候选待选操作闭环。
- 数据库差距：一期核心表大体齐全，`auto_check_candidate_files` 已进入后端候选文件流程；`file_parse_jobs`、`report_corrections` 等表尚未被完整业务流程使用。
- 领域模型差距：VDrive 候选文件对象已有后端 API 基础但未接前端闭环；报告更正还没有形成从 API、状态机到前端操作的闭环。

## 前端与业务方案对齐缺口

- 权限后台已接入账号搜索筛选、权限更新、停用/启用、删除访问配置、当前用户前端保护、会话同步、无权限只读提示和后台三栏/分区导航；业务数据范围 RBAC 已补齐，仍需补更完整前端行为测试。

- 报告页已支持列表和详情，但仍需补完整检查档案筛选、逐轮过程明细、报告更正入口、原始结论和更正后结论的追溯关系。
- 整改页已支持单项标记、撤销、待跟进落实和触发复查，但仍需补整改通知文案、批量操作、复查范围确认和二次确认。
- 工作台已具备最小进行中、复查中、待整改、待跟进、待归档入口；仍需补检查项/报告深链、超期推导和候选文件待选择入口。

1. 补更完整前端行为测试和 P0 体验细节：当前用户保护、会话同步、只读态、任务创建向导、归档确认、整改/复查确认，以及各业务页的权限态提示。

5. 再评估 `file_content`、`system_direct`、导出等 P2 能力。

## 使用规则

- 每次实现模块前，先读取对应模块文件的目标、前置依赖和验收标准，同时以本文件判断当前优先级。
- 每完成一个子任务，更新对应模块文件中的 checklist；若该 checklist 与本文件状态冲突，应同步修正或在本文件记录原因。
- 每完成一个模块，必须更新本文件中的后端/API 状态、前端/交互状态、业务方案对齐结论和下一步。
- P2 暂缓模块只有在 P0/P1 验收通过后再开始，或由用户明确指定提前启动。
