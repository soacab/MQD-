# CheckFlow MVP 任务进度

## 总体说明

本文件是 CheckFlow MVP 当前进度的总入口，用于对齐 `doc/tasks/` 模块任务、现有代码事实和 `企业办公_点检AI方案设计.md` 中的实际业务需求。

注意：本文件是当前状态总入口。执行具体模块前仍需读取对应任务文件；若模块 checklist 与本文件冲突，应以本文件的状态口径和最新验证结果为准，并在后续小任务中同步修正。

## 状态口径

| 状态 | 含义 |
| --- | --- |
| ✅ 已验证可用 | 已有实现和验证，能支撑当前定义的业务或工程目标。 |
| 🟡 骨架/最小链路已通，但未完全对齐业务方案 | 已有接口、页面或测试骨架，但距离实际业务流程、交互和验收标准仍有缺口。 |
| ⏳ 未开始 | 尚未进入实现，或只有任务文档无可用实现。 |
| ⏸ 暂缓 | 设计上确认暂不进入当前 MVP 优先级。 |

## 当前事实

- 后端 P0 主链路已有 API 和 unittest 覆盖，可以演示从项目、规则、任务、检查项确认、归档、报告、整改到复查的最小闭环。
- Phase 1 已明确保留轻量 SQL helper 路线；表结构、迁移、seed、集中枚举、核心唯一约束测试和 Alembic offline SQL 生成已验证可用，SQLAlchemy ORM Model 不作为当前 MVP Phase 1 验收项。
- 后端实现目前集中在 `backend/app/main.py`，后续继续扩展前应考虑按领域拆分服务、路由和仓储，避免主文件继续膨胀。
- 自动检查当前仍是 mock：`file_existence` 会写入模拟结果，不代表真实 VDrive 扫描、候选文件匹配或异常降级已经完成。
- 权限审查发现的业务数据范围 RBAC 已补齐并通过后端测试验证：`inspection_engineer` 仅能访问本人创建的点检任务及关联项目/派生数据，`project_admin` 可访问全量业务数据，组合权限取并集，权限管理不默认获得业务范围。
- 2026-06-30 提交前复核：后端 `uv run python -m unittest discover -s backend/tests -p 'test_*.py' -v` 40 项通过；前端 `npm test` 10 项通过；`npm run build` 通过。测试集已扩展到权限边界、检查档案作用域和前端防重复提交。
- 前端 P0 页面已接入登录、后台、项目、规则、点检、报告、整改复查等现有业务 API，可以支撑最小人工操作闭环；完整体验和真实自动检查仍需继续补齐。
- `企业办公_点检AI方案设计.md` 中的工作台分区已接入最小待办入口；候选文件确认、完整三栏点检体验、归档弹窗、整改通知文案、复查范围弹窗、复杂检查档案筛选等体验仍需继续补齐。

## Phase 2 复核结论

Phase 2 当前不是 ✅ 完成，而是 🟡 最小链路可用、完整验收未完成。`doc/tasks/02-auth-user-permission.md`、`doc/tasks/03-project-management.md`、`doc/tasks/04-vdrive-link-validation.md` 中的历史 checklist 暂不能作为完成依据，当前真实状态以本文件为准。

- 已具备：项目创建、VDrive mock 校验、项目列表和详情、加单、软删除，以及 `deleted` 项目阻断创建点检任务，能够支撑 P0 最小主流程。
- 已接通：用户权限后台支持权限更新、启用、停用、删除访问配置、权限筛选和无权限只读提示；“权限管理”不再自动获得点检、规则或项目管理能力。
- 已补齐：业务数据范围 RBAC 已覆盖本人范围、项目管理全量范围、组合权限取并集和权限管理无业务范围。
- 未完整：项目历史信息回填、QG 节点推荐、两步确认向导和真实 VDrive 可访问性校验仍未完整。
- 边界说明：VDrive 当前仍是 mock URL 解析和模拟返回，已有可替换 Adapter 边界和任务创建前校验动作，但不代表真实路径可访问校验已经完成。

## 与五份核心设计文档的主要差距

- 架构差距：后端实现仍集中在 `backend/app/main.py`，尚未按系统架构设计拆分 ProjectService、RuleService、InspectionService、ReportService、路由和仓储边界。
- API 差距：缺少 VDrive scan、扫描批次详情、候选文件选择/列表、自动检查重试、报告更正和报告导出等接口；部分已实现接口路径与设计稿仍有差异。
- 状态机差距：自动检查只有 mock success 主路径，缺少 `candidate_waiting`、not_found、多候选、扫描失败、重试等状态和业务动作。
- 数据库差距：一期核心表大体齐全，但 `auto_check_candidate_files`、`vdrive_scan_batches`、`vdrive_folders`、`vdrive_files`、`report_corrections` 等表尚未被完整业务流程使用。
- 领域模型差距：报告更正、VDrive 快照、候选文件对象还没有形成从 API、状态机到前端操作的闭环。

## 使用规则

- 每次实现模块前，先读取对应模块文件的目标、前置依赖和验收标准，同时以本文件判断当前优先级。
- 每完成一个子任务，更新对应模块文件中的 checklist；若该 checklist 与本文件状态冲突，应同步修正或在本文件记录原因。
- 每完成一个模块，必须更新本文件中的后端/API 状态、前端/交互状态、业务方案对齐结论和下一步。
- P2 暂缓模块只有在 P0/P1 验收通过后再开始，或由用户明确指定提前启动。
